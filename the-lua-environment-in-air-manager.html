The LUA Environment in Air Manager

  • Cr4O13
  • December 11, 2021
  • Published in: Air Manager

Air Manager is an application which allows to create your own 2D instrument panels for a variety of flight simulators. For the application user it provides a scripting environment through embedding the LUA programming language.

Each instrument script code is basically contained in a file named logic.lua. LUA calls it a chunk but we stay with script throughout this post.

The Instrument Script runs in a Sandbox

Each instrument script is executed in its own environment. Any bug/error in one instrument cannot affect other instruments. On the other hand, it means you cannot easily share state/data between instruments. This is called sandboxing.

This environment is created by Air Manager, before it starts to execute the instrument script.

What is The Content of the Environment?

Each script can only use what the environment provides. Air Manager on the one hand restricts what a plain LUA system would provide. On the other hand, it provides additional features that would not be available in a plain LUA system.

Basically, the instrument environment provides

  • the LUA version info,
  • most LUA built-in basic functions,
  • the functionality from some LUA standard libraries,
  • the Air Manager API functionality and
  • additional helper library functions

The version info is assigned to the global variable _VERSION.

You can get the version info through printing the variables value in the script:

print(_VERSION)

The value shown in the Air Manager 4.1 Console view is Lua 5.3.

LUA Built-In Functions

Air Manager provides all LUA built-in functions except

  • require()

The function require() is disabled as it depends on the package library which is not available. See below.

Restricted Access to LUA Standard Libraries

Air Manager provides only access to functions from the LUA standard libraries

  • os,
  • string,
  • math,
  • table and
  • coroutine.

Air Manager does not support access to the LUA standard libraries

  • io,
  • package,
  • debug and
  • utf8.

The io library would allow access to the device/file system. To achieve platform independence, Air Manager does not include the library in the environment. Instead, it provides a restricted set of functions for persisting state to and loading specific data formats from defined storage locations.

The package library is also not available because it would allow to load external library code and therefore circumvent the restrictions given above. But it means it is not possible to build modular scripts, using the require statement.

The debug library is not available as well. Probably because it would allow access to the internals of Air Manager. As an alternative Air Manager provides an integrated debugger in its IDE.

Then utf8 is also not supported. I assume because it is not required in a closed system like Air Manager.

The Air Manager API

Air Manager provides an extensive API. It comprises almost 300 functions covering all aspects of instrument building, communication with the simulators, support for hardware devices like Arduino, Raspberry Pico, ESP32 and more.

There is a extensive documentation available on the SimInnovations WIKI API page.
