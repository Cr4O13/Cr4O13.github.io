Give Variables a Meaningful Name

  • Cr4O13
  • December 12, 2021
  • Published in: Best Practices

In LUA variables are just places in memory that store values. To use the values in your code you have to give the places a name, some call them identifiers but we stick to the term name throughout this post.

Variables Have Names

The names can be any string of letters, digits, and underscores. But that string must not begin with a digit.

i, b10, index, MAXCOUNT, showImage, number_list, c2d2, _selected, _

are all valid names for variables.

A name starting with an underscore followed by one or more upper case letters, e.g. _ENV, should be avoided, as they are used by LUA itself.

The single underscore _ is a valid name but used as dummy variable by convention.

Names Are Case Sensitive

LUA is case-sensitive. That means the 3 names

maxcount, MaxCount, MAXCOUNT

identify three different variables. So be consistent with your style of naming variables.

Choose a Name With Meaning

A meaningful name gives more information to the reader of your code. Even if you are the only person ever to read through the code, you will be spending more time reading your code, than writing it.

After some time you will not remember what the purpose of that variable with the cryptic name is. The time you save when writing a cryptic name is not worth the time you need to repeatedly read through the code just to understand what it does.

What Is a Meaningful Name?

For a variable, a name is meaningful if it provides additional information that matches the context in which it is used.

If you are writing the code for a fuel level gauge (the context), a name that does not relate to the measuring of fuel tank levels or to the display of the level on the instrument, will be without meaning.

local tl_cur = 0
local tl_cap = 0
local tr_cur = 0
local tr_cap = 0

Those names do not convey any specific meaning in the context of a fuel gauge.

But those will:

local fuel_tank_left_level = 0
local fuel_tank_left_capacity = 0
local fuel_tank_right_level = 0
local fuel_tank_right_capacity = 0

An even better practice would be to use tables (see my other post on data structures) to maintain such structured data.

local fuel_tank = {
    left = { capacity = 0, level = 0 },
    right = { capacity = 0, level = 0 }
}

and access it using the dot operator .

 if fuel_tank.left.level >= 0.1 then ...

You may notice, that I always use variables declared as local. Why should you do that as well? See my other post on ‘Using Local Variables’.
